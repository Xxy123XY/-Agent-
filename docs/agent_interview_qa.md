# 求职辅助多智能体 Agent 系统面试问答

## 1. 你这个项目主要解决什么问题？

这个项目主要解决求职场景里几个割裂的问题：岗位 JD 信息分散、简历优化不够针对、面试准备缺少上下文、面试题生成成本高。  
我把这些能力整合成一个多智能体系统，让它可以围绕同一份 JD 和简历，完成岗位分析、简历优化、模拟面试和结果评估。

---

## 2. 为什么要做成多 Agent，而不是一个大 Agent？

因为这几个任务的职责差异很大。  
JD 分析关注信息抽取，简历优化关注生成，Reviewer 关注评估，Interviewer 关注出题和多轮对话。  
如果全放进一个 Agent 里，Prompt 会很重，状态也会混乱，后期很难维护。  
拆成多 Agent 后，每个节点只负责一类任务，流程更清晰，也更方便调试和扩展。

---

## 3. 你的项目里有哪几个核心 Agent？

核心有五个：

- `JDAnalyzer`：把原始 JD 解析成结构化结果
- `Writer`：根据 JD 和用户要求优化简历
- `Reviewer`：对优化后的简历进行审查和反馈
- `Interviewer`：生成面试题、进行模拟面试、输出评估
- `Supervisor`：决定当前任务是否要调用题库、RAG 或 web_search

这些 Agent 通过 LangGraph 和状态传递连接起来。

---

## 4. LangGraph 在这个项目里起什么作用？

LangGraph 主要负责流程编排。  
我把各个 Agent 节点串成一个有状态工作流，让结构化 JD、简历版本、RAG 上下文、评审反馈这些信息可以在节点之间持续传递。  
相比普通函数串联，LangGraph 更适合描述多步骤、多状态、可回退的 Agent 流程。

---

## 5. 你为什么设计了 Supervisor Agent？

因为我发现如果每次都固定走“题库检索 + web_search + LLM 生成”，成本会比较高，而且有些步骤是没必要的。  
所以我加了一个轻量 Supervisor，根据题库命中数量、是否有 RAG 上下文、JD 技能信息是否充分，动态决定是否启用题库、RAG 和 web_search。  
这样可以减少重复调用，提升整体效率。

---

## 6. Supervisor 是怎么做决策的？

目前是规则路由，不是再额外调用一次 LLM。  
比如在面试题生成场景下：

- 如果题库命中数量足够，就优先使用题库，跳过 web_search
- 如果题库命中不足，再启用 web_search 补充真实面经
- 如果当前有本地 RAG 上下文，就一并纳入出题参考

这样做的原因是决策逻辑比较明确，用规则更稳定，也不会增加额外模型成本。

---

## 7. 你为什么要把 Agent 和 RAG 解耦？

因为 Agent 的职责应该是推理、规划和工具调度，而不是把文档分块、向量化、检索、融合这些底层细节都耦合进去。  
我把知识库能力抽成独立 RAG 模块，Agent 只需要调用检索结果。  
这样后续如果要替换 embedding 模型、修改检索策略或者更换向量库，不需要改 Agent 主逻辑。

---

## 8. 你的 RAG 是怎么实现的？

当前 RAG 主要分几步：

1. 读取知识库文档
2. 按 chunk size 和 overlap 切分
3. 用 embedding 模型做向量化
4. 存入 ChromaDB
5. 查询时同时做向量检索和关键词检索
6. 最后用 RRF 做融合排序

检索结果会组织成可直接注入 Prompt 的上下文，同时也会把命中文档展示到前端。

---

## 9. 为什么要做混合检索，而不是只做向量检索？

因为只做向量检索会漏掉一些关键词很强、但语义相似度不够高的内容。  
尤其是在 JD、简历模板和面试题这种场景里，像技术名词、缩写、岗位关键词都很重要。  
所以我用了 Dense + Sparse 双路召回，让语义和关键词两种信号都能参与排序，整体效果更稳。

---

## 10. 你的 Dense 和 Sparse 检索分别是什么？

Dense 检索是基于 embedding 的向量相似度检索，底层用的是 ChromaDB。  
Sparse 检索是本地 TF-IDF 关键词检索。  
最后我用 RRF 融合两个结果，而不是直接依赖某一路。

---

## 11. 为什么你用了 TF-IDF，没有直接用 BM25？

一方面是当前项目更偏原型和应用落地，TF-IDF 接入成本低，实现简单，已经能覆盖大部分关键词召回需求。  
另一方面它和现有本地检索流程兼容性比较好。  
如果后续继续迭代，我觉得可以把 TF-IDF 升级成 BM25，这样在长文本和词频区分上会更稳。

---

## 12. RRF 是什么，你为什么用它？

RRF 是一种结果融合方法，全称 Reciprocal Rank Fusion。  
它不直接比较不同检索器的原始分数，而是根据各自排序位置做融合。  
我用它的原因是向量检索和关键词检索的分数体系不一致，直接加权不稳定，RRF 更简单也更鲁棒。

---

## 13. 你的知识库里主要存了什么？

主要存三类内容：

- 岗位相关术语和知识
- 简历模板和优秀表达
- 面试准备资料

这些内容主要服务于 JD 分析、简历优化和面试出题。

---

## 14. 为什么还要单独做一个面试题向量题库？

因为普通知识库更偏“资料型内容”，而面试题本身是一类更适合复用和沉淀的数据。  
我把历史生成题和联网搜索到的真实面经题单独存成题库，这样后续遇到相似岗位时，可以先检索已有高频题，而不是每次重新生成。

---

## 15. 面试题题库里存了哪些信息？

每道题除了题目正文，还会存一些 metadata，比如：

- `question_id`
- `category`
- `topic`
- `source`
- `expected_points`
- `quality_score`
- `used_count`
- `created_at`
- `last_used_at`

这样后续不仅能检索，还能做来源标注、使用统计和清理策略。

---

## 16. web_search 为什么要回写题库？

因为 web_search 获取的是一次性的外部信息，如果不沉淀，下次遇到类似岗位还要重新搜索。  
我把其中抽取出来的题目写回向量题库，相当于把外部信息转成系统内部可复用资产。  
这样题库会随着使用不断增长，后续生成题目时能更多依赖本地题库，减少重复联网和重复生成。

---

## 17. 题库为什么要做阈值清理？

因为题库如果无限增长，检索噪声会变大，维护成本也会增加。  
所以我设置了最大容量阈值，当题库超过上限时，会优先清理低质量、低使用、较旧的问题。  
这样可以让题库保持相对高质量。

---

## 18. 你的简历优化流程为什么要分成 Writer 和 Reviewer？

因为生成和评估本质上是两类不同任务。  
Writer 负责根据 JD 和用户需求生成内容，Reviewer 负责检查关键词覆盖、表达质量和岗位匹配度。  
这样可以形成一个简单的反思闭环，避免一次生成直接输出，提升稳定性。

---

## 19. JD 分析模块具体做了什么？

它会把原始 JD 解析成结构化字段，包括：

- 核心技能
- 硬性要求
- 软性能力
- 业务关键词
- 岗位总结

这样后面的简历优化、匹配度计算和面试出题都不需要反复解析原始 JD。

---

## 20. State 在你的系统里起什么作用？

State 是多 Agent 协作的上下文载体。  
我用它在节点之间传递结构化 JD、原始简历、优化后简历、RAG 上下文、面试题、评审结果、执行轨迹等信息。  
它的作用是让流程不是“每一步从零开始”，而是共享同一个任务上下文。

---

## 21. 你是怎么处理 Agent 输出不稳定的问题的？

我加了一层统一的结构化解析逻辑。  
因为大模型有时会多输出说明文字、代码块，或者 JSON 格式不完全规范，所以我做了兜底解析，包括清理多余内容、补全边界、给默认字段等。  
这样可以减少因为输出格式问题导致的流程中断。

---

## 22. 你项目里遇到过哪些比较典型的问题？

比较典型的有几个：

- embedding 连接失败
- Streamlit 的 session_state 更新时机报错
- Chroma 在 metadata 更新时触发 embedding 维度冲突
- Agent 输出 JSON 偶尔不规范

这些问题后来我都分别通过环境排查、状态管理调整、只更新 metadata、增加解析层的方式解决了。

---

## 23. 为什么会出现 embedding connection error？你怎么排查的？

主要从三个方向排查：

1. 模型和接口地址是否匹配
2. chat model 和 embedding model 是否混用了错误配置
3. 当前运行环境是不是正确的 Python 环境

后来我把 chat 和 embedding 的配置分开管理，并且检查 `.env`、base_url、模型名和运行环境，问题基本定位清楚了。

---

## 24. 你为什么不让 Supervisor 也用 LLM 做决策？

因为目前的决策条件比较明确，比如题库命中数、是否有 RAG 上下文、是否需要联网搜索。  
这种场景下规则已经足够，而且规则路由更可控、更稳定，也不会增加额外调用成本。  
如果以后任务复杂度更高，再考虑升级成 LLM + 规则混合决策。

---

## 25. 你怎么证明这个系统比单轮生成更好？

主要有三个方面：

1. 架构上更清晰：不同任务拆成独立 Agent，更容易维护和扩展
2. 效果上更稳：加入 RAG、Reviewer 和题库复用后，结果更有上下文，不容易完全泛化
3. 成本上更低：Supervisor 会在题库命中足够时跳过 web_search，题目也能复用，不需要每次都从零生成

虽然目前没有做非常严格的量化 benchmark，但从功能稳定性和结果质量上已经比单轮生成更合理。

---

## 26. 如果继续优化这个项目，你下一步最想改什么？

我会优先做两件事：

1. 把当前的关键词检索从 TF-IDF 升级成 BM25，进一步提升混合检索效果
2. 增加更完整的运行观测能力，比如节点级耗时、错误分类和调用链记录

前者更偏效果优化，后者更偏工程化落地。

---

## 27. 你觉得这个项目里最有价值的设计点是什么？

我觉得有两个最有价值：

第一是把求职场景拆成多 Agent 工作流，而不是单个大 Prompt；  
第二是把面试题做成可沉淀、可复用、可动态路由的向量题库。  

前者体现的是系统设计，后者体现的是把一次性生成结果变成长期资产。

---

## 28. 如果面试官问“这个项目最像工程而不是 Demo 的地方是什么”，你怎么回答？

我会说有三点：

- 做了明确的职责拆分，Agent、RAG、题库、工具层不是全耦合在一起
- 做了混合检索和题库沉淀，不是单纯调一次大模型接口
- 做了动态路由和状态传递，流程是可编排、可扩展的

这些设计让项目不只是一个聊天页面，而是一个围绕求职任务构建的 Agent 应用系统。

---

## 29. 如果面试官追问“你个人贡献最大的是哪部分”，怎么回答？

可以重点说三块：

1. 多 Agent 架构拆分和 LangGraph 流程编排
2. RAG 混合检索和面试题向量题库设计
3. Supervisor 路由逻辑和题库回写机制

因为这三部分最能体现你不是只做了前端拼装，而是真正参与了核心架构设计和功能实现。

---

## 30. 一句话总结这个项目，你会怎么说？

这是一个基于 LangGraph 的求职辅助多智能体系统，我把 JD 分析、简历优化、模拟面试和面试题题库沉淀串成了一个可编排、可检索、可复用的 Agent 工作流。

---

## 31. 你的 Agent 记忆模块是怎么设计的？

当前项目里的记忆模块是长期记忆，不是短期对话缓存。  
我单独做了一个 `MemoryManager`，底层还是基于 ChromaDB，把每次面试结束后的总结摘要、评分结果和时间戳存成向量化记忆。  
下次再进入面试环节时，会根据当前 JD 或面试主题检索相似历史记录，再把这些记忆注入到面试官的 System Prompt 里，帮助它做更连续的提问和评估。

---

## 32. 为什么你把记忆模块设计成独立组件？

因为记忆本质上也是一种可检索上下文，不应该直接写死在某个 Agent 里。  
我把它抽成独立的 `MemoryManager`，和 RAG、题库一样都作为基础能力层。  
这样后续如果要更换存储方式、调整记忆检索策略，或者把记忆接入更多 Agent，不需要改主流程逻辑。

---

## 33. 你现在的记忆模块有哪些不足？

现在的实现更像一个基础版本，主要有三个不足：

- 还没有稳定的用户标识体系，严格来说还不能精准判断“是不是同一个候选人”
- 记忆保存的是总结摘要，不是更细粒度的行为片段
- 检索策略还比较简单，主要是向量相似度，没有做时间衰减或优先级控制

如果继续优化，我会补充候选人 ID、记忆分层和更细的召回策略。

---

## 34. RAG 效果怎么样？你是怎么评估的？

目前我对 RAG 的评估主要分成两层：

第一层是功能层面，看它能不能稳定召回与岗位、简历和面试主题相关的内容；  
第二层是结果层面，看引入 RAG 后，简历优化建议和面试题是否比纯 LLM 生成更贴近岗位要求。  

当前项目还没有做完整的自动化 benchmark，但我会通过查询样例、命中文档可视化、生成结果对比来做人工评测。  
从实际效果看，混合检索相比单一路径召回更稳，尤其在关键词很强的场景下更明显。

---

## 35. 如果要正式评测 RAG，你会怎么做？

我会从离线评测和在线评测两部分做：

离线评测主要看检索质量，比如：

- Recall@K
- Precision@K
- MRR
- 命中是否覆盖核心技能词

在线评测主要看生成质量，比如：

- 简历优化内容是否覆盖 JD 关键要求
- 面试题是否和岗位技能、候选人经历相关
- Reviewer 或人工标注对结果的接受度

如果数据量够，我会构建一批标准问答或标准查询集做对比测试。

---

## 36. 你怎么提升 RAG 的召回率和准确率？

召回率和准确率通常是一起权衡的，我会分开看：

提升召回率的方式：

- 优化分块策略，避免 chunk 过大或过碎
- 增加关键词检索，让技术名词和缩写不容易漏掉
- 扩大召回候选数，再做融合排序

提升准确率的方式：

- 用 RRF 融合 Dense 和 Sparse 结果，减少单一路径噪声
- 优化 query 构造，让检索更贴近 JD 和简历语境
- 给文档增加更丰富的 metadata，便于筛选
- 在生成前只保留高相关结果，减少噪声注入 Prompt

在我的项目里，混合检索和 Supervisor 路由就是这两个目标之间的一个平衡。

---

## 37. 你怎么判断一次检索结果“好不好”？

我一般看三个维度：

- 命中的内容是否真的和当前岗位或问题相关
- 命中结果里是否覆盖了核心技能词和关键业务词
- 这些检索结果注入后，最终生成是否明显变得更具体、更有针对性

如果检索命中了很多文本，但生成结果还是泛，那通常说明召回噪声太高，或者 query 构造还有问题。

---

## 38. 你的 MCP 服务具体是怎么设计的？

我在项目里单独写了一个 `MCP Server`，用 `FastMCP` 暴露工具能力，通过 `stdio` 和客户端通信。  
服务端主要封装了几类求职场景工具：

- 行业术语检索
- 简历模板检索
- 简历评分
- 面试题库检索
- web_search

客户端这边通过 `langchain-mcp-adapters` 启动 MCP 子进程，并把这些工具转成 LangChain Tool，供 LangGraph 或 Agent 直接调用。

---

## 39. 你为什么要用 MCP，而不是直接在代码里调本地函数？

直接调本地函数当然也能跑，但 MCP 的价值是标准化和解耦。  
我把工具能力单独封装成 MCP 服务后，Agent 不需要关心工具内部实现，只需要按照协议调用。  
这样后续如果要替换工具实现、迁移到别的客户端，或者扩展更多工具，改动会更小。

---

## 40. 你的 MCP Client 是怎么工作的？

当前实现里，Client 会在第一次调用时启动 MCP Server 子进程，然后通过 `MultiServerMCPClient` 建立 `stdio` 连接。  
拿到的工具会被包装成 LangChain Tool，Agent 可以直接绑定和调用。  
如果 MCP 启动失败，还会回退到本地工具实现，避免整条链路不可用。

---

## 41. 你这个项目里有哪些现成的 MCP 工具？

目前已经封装好的主要有五个：

- `search_industry_terms`
- `get_resume_template`
- `evaluate_resume_score`
- `get_interview_question_bank`
- `web_search`

它们分别服务于 JD 理解、简历优化、简历评估和面试准备这些场景。

---

## 42. 如果继续扩展 MCP，你还会加什么工具？

我觉得比较有价值的有四类：

- `resume_diff`：对比原始简历和优化简历的变化
- `job_match_score`：更细粒度地评估简历和 JD 的匹配度
- `question_dedup`：对面试题做去重和聚类
- `company_research`：针对目标公司做岗位、技术栈、业务背景检索

这样 MCP 不只是一个工具集合，还能逐渐演化成求职场景的能力层。

---

## 43. 你觉得记忆、RAG、题库、MCP 这几个模块之间是什么关系？

我会把它们看成 Agent 下面的四类基础能力：

- RAG：负责通用知识检索
- 题库：负责结构化、可复用的面试题资产
- 记忆：负责跨会话保留候选人历史表现
- MCP：负责把这些能力和外部工具标准化暴露给 Agent

真正做决策的是 Agent，本质上它们都是 Agent 的上下文和工具层。

## 44. 如果面试官问“你项目里最值得继续深挖的部分是什么”，你怎么回答？

我会说有三个方向：

1. 记忆模块做用户级身份绑定和分层记忆

3. MCP 工具继续扩展成更完整的求职能力服务层
